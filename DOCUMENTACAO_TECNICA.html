1️⃣ Arquitetura e Padrões

O sistema foi desenvolvido seguindo os princípios de:

  • Domain-Driven Design (DDD): Organização baseada em domínios de negócio
  • Clean Code: Código limpo e bem estruturado
  • Repository Pattern: Abstração de dados
  • Service Layer: Lógica de negócio centralizada
  • Value Objects: Encapsulamento de valores com regras

3️⃣ Domain-Driven Design (DDD)

Entidades (Entities)

Objetos com identidade única que encapsulam comportamento e estado:

Entidade Arquivo Responsabilidades
User Domain/User/User.php Gerenciar dados do usuário, saldo, 2FA
Document Domain/Document/Document.php Gerenciar documentos, versões, status
Category Domain/Category/Category.php Categorias de documentos, estatísticas
Payment Domain/Payment/Payment.php Gerenciar pagamentos, status

Value Objects

Objetos imutáveis que representam valores com regras de negócio:

Value Object Validações Métodos
UserId String não vazia getValue(), equals()
Email Formato de email válido getValue(), equals()
Password Mín. 8 caracteres hash(), verify()
UserType Tipo válido (ADMIN, PROFESSOR, etc) isAdmin(), isProfessor(), canUploadFree()

8️⃣ Segurança

Autenticação

  • Senhas com hash BCRYPT (cost: 12)
  • JWT com expiração (24h)
  • 2FA com código de 6 dígitos
  • Session timeout

Validação

  • Validação no Value Object (client-side)
  • Validação na Entidade (regras de negócio)
  • Validação no Repositório (constraints de BD)
  • Validação no Endpoint (input)

Proteção de Dados

  • PDO com prepared statements
  • CORS configurado
  • Headers de segurança (X-Content-Type-Options, etc)
  • Links temporários com expiração
